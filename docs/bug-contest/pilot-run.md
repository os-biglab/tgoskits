# 校内试运行方案

本试运行用于验证两件事：

- 这套找 Bug 比赛题库对大学生是否有合适的难度梯度。
- 官方镜像、统一判题入口和 hidden tests 设计是否足够稳定。

结构化题单见 `pilot-batch.yaml`。

## 1. 试运行范围

首批试运行采用 `6 道 Host 题 + 2 道 ArceOS 题` 的最小闭环：

| 类型 | 题目 ID | 目的 |
| --- | --- | --- |
| Host | `host-memory-allocators-bundle` | 验证新手是否能快速进入系统组件题 |
| Host | `host-sync-timer-bundle` | 验证并发与计时类题的梯度 |
| Host | `host-io-access-bundle` | 验证网安学生对接口边界与权限题的适配度 |
| Host | `host-axsched-dedicated` | 验证单组件专属题的区分度 |
| Host | `host-rsext4-dedicated` | 验证文件系统题的说明成本与 hidden tests 价值 |
| Host | `host-smoltcp-dedicated` | 验证协议解析题对网安方向学生的吸引力 |
| ArceOS | `arceos-axtask-dedicated` | 验证 QEMU 级并发题的可进入性 |
| ArceOS | `arceos-axfs-dedicated` | 验证块设备与文件系统链路题的稳定性 |

之所以不在首轮试运行引入 StarryOS 和 Axvisor，是为了先确认：

- Host 与 ArceOS 的基础体验是否顺畅
- 题面模板与提交格式是否足够清晰
- 评测队列和人工复核是否能承受预期流量

## 2. 参与规模

建议参与人数：

- `24-40` 名学生
- 允许个人参赛，也允许 `2` 人组队
- 优先邀请网络安全专业和计算机专业混合报名

建议的背景构成：

- 三分之一是有 OS 基础、但没看过 TGOSKits 的学生
- 三分之一是有 Rust 或系统编程经验的学生
- 三分之一是网络安全方向、擅长输入边界与攻击面分析的学生

这样更容易验证赛题是否真的适合跨专业学生。

## 3. 时间安排

建议采用 `2 天热身 + 5 天试运行 + 2 天复盘`：

### 3.1 热身期

- 发放官方镜像与最小文档阅读包
- 让学生完成 1 道样例 Host 题
- 确认他们知道如何使用 `repro.sh`、提交模板和评测入口

### 3.2 试运行期

- 第 1-2 天：开放 3 道 Host 审计包
- 第 3 天：开放 `host-axsched-dedicated` 与 `host-rsext4-dedicated`
- 第 4 天：开放 `host-smoltcp-dedicated`
- 第 5 天：开放 2 道 ArceOS 题

这样的节奏有两个好处：

- 先验证大部分学生能否顺利进入 Host 赛道
- 再确认 QEMU 级题目会不会让大量学生在环境上掉队

## 4. 成功指标

建议至少跟踪以下指标：

| 指标 | 目标 |
| --- | --- |
| 判题成功率 | `>= 95%` |
| flaky case 比例 | `<= 3%` |
| 单次反馈中位时间 | `<= 8 分钟` |
| visible/hidden tests 一致性 | `>= 90%` |
| Host 题解出率 | `40%-80%` |
| ArceOS 题解出率 | `20%-60%` |

额外观察项：

- 选手第一次拿到“有效提交”的时间
- 一道题平均重试次数
- 哪类 hidden tests 最容易把“投机补丁”筛出来
- 哪种题面最容易引起歧义

## 5. 数据采集

试运行期间建议记录：

- 每条提交的时间、题目、运行时长和结果
- visible tests 通过但 hidden tests 失败的具体模式
- 人工复核拒绝的原因
- 选手在根因分析里最容易误判的地方

如果条件允许，可以额外记录两类数据：

- 不同专业背景在不同题型上的表现差异
- 回归测试提交质量与最终得分的相关性

## 6. 现场支持机制

建议至少安排：

- 1 名平台值守，处理镜像、评测队列和账户问题
- 2 名系统方向助教，处理命令与仓库结构问题
- 1 名安全方向助教，处理输入建模与披露边界问题

试运行期间可以回答：

- 如何运行公开复现命令
- 如何理解目录结构与模板
- 如何提交 patch、测试和说明

不应回答：

- 根因在哪一行
- hidden tests 的具体内容
- 标准修复方式

## 7. 退出条件

试运行通过后，才建议进入完整比赛筹备。这里的“通过”指本轮 `8` 道试运行题里，至少 `6` 道能稳定跑通且具备可接受的解题分层：

- 至少 6 道题在官方镜像中稳定运行且不需要现场热修环境
- Host 题与 ArceOS 题都形成清晰分层
- 人工复核能稳定识别“只撞 visible tests”的投机补丁
- 至少产生 3 份值得回流到正式仓库的回归测试或复现脚本

如果未满足上述条件，优先收缩范围，而不是继续扩题：

- 如果 Host 题已经稳定，但 ArceOS 题波动大，就先冻结在 Host 赛道
- 如果文件系统题太难，可把 `host-rsext4-dedicated` 延后到正式预赛第二周
- 如果协议栈题日志过长，就优先改良 `repro.sh` 和题面，而不是删掉题目

## 8. 试运行后的产出

试运行结束后至少要产出三份材料：

1. 一份题目复盘，说明每题的解出率、争议点和是否继续保留。
2. 一份环境复盘，说明镜像、QEMU 与 hidden tests 是否稳定。
3. 一份质量回流清单，说明哪些提交值得沉淀回仓库。

如果这三份材料都能形成，说明比赛已经从“设计方案”进入“可执行项目”阶段。
