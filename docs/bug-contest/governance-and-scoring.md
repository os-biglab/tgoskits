# 治理与评分规范

本规范用于把比赛从“能出题”推进到“能稳定组织、能公平评审、能把成果回流到仓库”。
它和 `component-ledger.yaml`、`judge-manifest.yaml`、`challenge-packaging.md` 一起构成比赛运营的主文档。

## 1. 组织结构

建议设立 5 个角色层级：

| 角色 | 主要职责 | 人数建议 |
| --- | --- | --- |
| 总负责人 | 决定赛制、时间线、争议处理与赛后回流 | 1 |
| 赛道负责人 | 分别负责 `components/*`、ArceOS、StarryOS、Axvisor 四条线 | 4 |
| 出题人 | 负责单题设计、模板填写、visible/hidden tests 与标准答案 | 每题 1 |
| 复核人 | 对题目和选手提交做第二视角审核 | 每题 1 |
| 私密披露小组 | 处理真实漏洞披露、保密、是否转入补丁流程 | 3 |

执行规则：

- 每道题必须至少有 `出题人 + 复核人` 两人签字通过，才能进入正式赛。
- 真实漏洞披露赛道不能只由单一出题人或维护者裁决，必须经过私密披露小组复核。
- Axvisor 赛题在上线前需要额外完成一次环境干跑，因为镜像与配置耦合更强。

## 2. 标准题评分

标准题采用“难度基分 + 自动评测 + 人工复核 + 影响系数”四层计分。
为了避免“只撞 visible tests 就高分”，hidden tests 与人工复核必须在总分里占有明确权重。

```text
原始得分 = 难度基分 × (0.55 × visible_tests + 0.15 × hidden_tests + 0.30 × manual_review)
最终得分 = 原始得分 × 影响系数
```

说明：

- `visible_tests`、`hidden_tests`、`manual_review` 都按 `0.0 - 1.0` 折算。
- `manual_review` 由下文的 40 分人工复核表折算而来，即 `manual_review = 人工复核得分 / 40`。
- 这样设计后，hidden tests 占总分 `15%`，人工复核占总分 `30%`，两者合计 `45%`，高于“至少 30%”的最低要求。

### 2.1 难度基分

| 难度 | 分值 | 适用题型 |
| --- | --- | --- |
| `L1` | 100 | Host 审计包、热身题、单一边界条件题 |
| `L2` | 200 | 预赛专属题、跨模块但仍可自动判的题 |
| `L3` | 300 | 决赛题、虚拟化题、强系统集成题 |

### 2.2 自动评测部分

自动评测拆为两部分：

| 部分 | 权重 | 说明 |
| --- | --- | --- |
| visible tests | 55% | 公开测试与基础功能门槛 |
| hidden tests | 15% | 防止只撞样例、覆盖额外边界 |

### 2.3 人工复核部分

人工复核总权重固定为 `30%`，内部采用 40 分制，再折算到总分。

40 分制建议拆成四项：

| 维度 | 分值 | 说明 |
| --- | --- | --- |
| 回归测试 | 15 | 是否补了真正能抓住同类回归的测试或复现脚本 |
| 补丁最小化 | 10 | 是否只改必要路径，没有大面积无关改动 |
| 根因分析 | 10 | 是否能准确解释问题成因与修复逻辑 |
| 影响分析 | 5 | 是否讲清组件层次与潜在波及范围 |

折算方式：

```text
manual_review = 人工复核得分 / 40
```

### 2.4 影响系数

| 条件 | 系数 |
| --- | --- |
| 仅影响单一组件、无额外安全边界 | `1.0` |
| 基础复用组件，可能同时影响多个系统 | `1.2` |
| 涉及安全边界、Guest/Host 隔离或高风险输入路径 | `1.3` |

## 3. 真实漏洞披露赛道评分

真实漏洞披露赛道不采用固定分值，而采用区间评分：

| 档位 | 分值区间 | 适用情况 |
| --- | --- | --- |
| `P1` | 200-300 | 可复现，但影响范围较小或修复不完整 |
| `P2` | 350-500 | 影响明确、修复可信、回归测试完整 |
| `P3` | 550-800 | 同时具备高价值影响、可合入补丁和高质量测试 |

评审维度：

- 是否稳定复现
- 是否是仓库真实问题而非题面故意缺陷
- 是否可能影响 ArceOS / StarryOS / Axvisor 之一或多者
- 是否给出可合入修复与回归测试
- 是否需要走更严格的私密修复流程

## 4. 评审与复核流程

建议每条标准题的提交流程如下：

1. 平台执行 visible tests。
2. 平台执行 hidden tests。
3. 平台执行 `cargo fmt --all -- --check` 与 `cargo xtask clippy`。
4. 若三步全部通过，进入人工复核。
5. 人工复核检查补丁范围、根因分析与是否存在投机行为。
6. 按 `visible_tests / hidden_tests / manual_review` 三段结果计算得分并入榜。

人工复核最低检查项：

- 是否修改了允许路径之外的内容
- 是否删除或绕过真实逻辑
- 是否只是针对 visible tests 定制返回值或分支
- 是否确实补了回归测试或等价复现脚本
- 根因分析是否与代码修改一致

## 5. 晋级规则

推荐预赛晋级机制：

- 排名前 20-30 支队伍进入人工复核池。
- 通过人工复核后再确定决赛名单。
- 若两支队伍自动分相同，按以下顺序打破平局：
  - hidden tests 通过率
  - 人工复核得分
  - 高难题得分
  - 首次正确提交时间

## 6. 申诉规则

必须事先定义申诉规则，避免比赛中出现临时口径：

- 选手可对题面歧义、评测异常、hidden tests 误判提出申诉。
- 申诉窗口建议为每次评分结果公布后 24 小时内。
- 申诉必须附最小复现信息，不能只说“我觉得不公平”。
- 涉及标准答案和 hidden tests 的争议，由对应赛道负责人和复核人共同裁决。

## 7. 反作弊与学术诚信

建议提前写入比赛规则：

- 禁止在公开平台发布未修复漏洞细节。
- 禁止抄袭其他队伍的补丁、说明和回归测试。
- 禁止通过删除测试、篡改评测脚本或修改不允许路径来绕过判题。
- 禁止利用比赛平台的联网能力做与题目无关的外部访问。

平台侧建议：

- 默认断网运行。
- 保留每次提交的 diff、日志和产物哈希。
- 对高分提交做相似度检查，重点看 patch 结构和说明文本。

## 8. 私密披露流程

真实漏洞披露赛道必须采用私密通道，推荐流程如下：

1. 选手提交私密报告、补丁和回归测试。
2. 平台自动通知私密披露小组。
3. 小组在 72 小时内确认是否属于真实问题、是否需要保密升级。
4. 若需要保密升级，则暂停公开计分，仅记录受理状态。
5. 完成修复与合入窗口后，再决定公开积分、署名和赛后披露方式。

保密分级建议：

| 分级 | 说明 | 处理方式 |
| --- | --- | --- |
| `S1` | 稳定性问题，无明显安全边界影响 | 可按普通高分题处理 |
| `S2` | 可能影响隔离、权限或 guest/host 边界 | 进入私密修复流程 |
| `S3` | 明显高危且可被外部利用 | 只允许极小范围披露，延后公开 |

## 9. 奖项设置

建议同时设置“总排名奖”和“专项奖”，避免只奖励刷题速度：

- 综合一等奖、二等奖、三等奖
- 最佳系统实现奖：补丁最扎实、回归测试最完整
- 最佳安全发现奖：安全影响最大、边界分析最清晰
- 最佳测试贡献奖：沉淀测试价值最高
- 最佳跨专业协作奖：网安与计科协作最强

## 10. 赛后回流规则

比赛结束后，至少要完成以下 4 项收口：

1. 把可合入的修复分发给对应赛道负责人和仓库维护者。
2. 把高价值回归测试吸收到正式测试入口。
3. 统计本赛季覆盖组件数、合入补丁数、回归测试新增数。
4. 产出赛后报告，说明哪些题真正提升了仓库质量。

## 11. 推荐的评审边界

为了避免“自动评测一过就算完成”，建议把 hidden tests 与人工复核明确写进评分公式，而不是只作为口头要求。
本规范中：

- hidden tests 固定占总分 `15%`
- 人工复核固定占总分 `30%`

人工复核至少包括：

- 代码修改是否准确
- 回归测试是否有效
- 根因分析是否可信
- 是否值得回流进正式仓库

这样既能保持自动判题效率，又能防止比赛退化成“撞 hidden tests 大赛”或“只靠人工裁量大赛”。
